## 1. iOS PWA Web Push API

### Version Requirements
- **Minimum:** iOS 16.4 / Safari 16.4 (released March 2023)
- **Mechanism:** Web Push is ONLY available for PWAs added to the home screen from Safari — it does NOT work in in-browser tabs
- **Manifest requirement:** `"display": "standalone"` is mandatory in the web app manifest; without it, iOS will not expose the Push API

### EU / Regional Constraints
- Apple initially announced removing PWA push support in the EU for iOS 17.4 (DMA compliance), but **reversed this decision** after developer backlash
- As of 2026, EU users retain full PWA push notification support
- However, EU regulation continues to evolve — monitor for future changes

### Known Reliability Issues (2025–2026)
- **Critical reliability problem:** Multiple developers report that push notifications work for a few days, then **silently stop being delivered**. This is a known, unresolved Safari WebKit bug. It does not affect the App Store version or Android
- **No Background Sync/Fetch:** iOS does not support Background Sync API, Periodic Background Sync, or Background Fetch for PWAs — no silent push / data-only notifications possible
- **No rich push features:** No badge count control, no rich media attachments, no background processing triggers
- **Opt-in and delivery rates:** Web push opt-in rates (~33% delivery reliability) are significantly lower than native push (~95%+)
- Safari 18.4 (2025) added Declarative Web Push, a slight improvement but does not fix the reliability issues

### Feasibility & Recommendation
- **Feasible** for basic notification delivery on iOS 16.4+, but **unreliable for production** due to the silent-stop bug
- **Recommended pattern:** Use Web Push as a best-effort channel; implement an in-app notification fallback (e.g., polling or WebSocket-based notification badge) for critical messages
- For a couples app where timely delivery matters, plan for native (Capacitor/iOS) alongside the PWA path

## 2. Django Channels WebSocket JWT Authentication

### The Core Constraint
The JavaScript `WebSocket` API **does not support custom HTTP headers** (like `Authorization: Bearer <token>`). This makes standard Bearer token auth impossible during handshake.

### Standard Patterns

#### Pattern A: JWT in Query Parameter
```
ws://host/ws/chat/?token=<access_jwt>
```
- Most common approach
- Middleware parses `scope['query_string']`, extracts token, validates, populates `scope['user']`
- Libraries: `django-channels-jwt-auth-middleware` (PyPI), manual middleware gist by rluts

**Security risk:** Token appears in URL — can leak via server logs, browser history, proxy caches

#### Pattern B: UUID Ticket / One-Time Token (Recommended)
1. Client calls a REST endpoint (authenticated via HttpOnly cookie or Bearer) to get a short-lived UUID "ticket" (stored in Django cache, TTL ~30s)
2. Client opens WS: `ws://host/ws/chat/?ticket=<uuid>`
3. WS middleware validates UUID against cache, retrieves user, deletes ticket

Implemented in `django-channels-jwt` (PyPI). Avoids exposing JWT in URL.

#### Pattern C: Cookie-based (HttpOnly JWT)
- WebSocket handshake IS an HTTP request — cookies are sent automatically
- If access token is stored in an HttpOnly cookie, Django Channels middleware can read it from `scope['headers']` or via `scope['cookies']`
- **CSRF risk:** Any origin can initiate a WS connection to your domain using the user's cookies — must validate `Origin` header in the WS consumer
- Channels docs recommend checking `ALLOWED_HOSTS` and implementing origin validation

### HttpOnly Refresh Token Concern
- Refresh tokens in HttpOnly cookies work for REST endpoints but are **awkward for WS flows** since the WS connection doesn't support header-based token refresh mid-stream
- Pattern: Use short-lived access JWT in query param for WS handshake; refresh via a REST call before reconnecting when WS drops

### Recommended Implementation
1. Store refresh token in HttpOnly Secure cookie (REST endpoint manages rotation)
2. Generate short-lived access JWT on login; pass via query param OR use UUID ticket pattern for WS
3. Implement `JWTAuthMiddlewareStack` wrapping `URLRouter` in `asgi.py`
4. Validate `Origin` header in every consumer's `websocket_connect` handler

## 3. Capacitor + FCM for Android (Sideloaded APK)

### Current Capacitor Version
- **Capacitor 8.3.0** is the current stable release as of early 2026 (not v5 or v6 — those are older)
- Capacitor 6 was a major release; v7 and v8 followed. Migration guides exist for each major version
- `@capacitor/push-notifications` current version: **7.0.6** (tracks Capacitor major version)

### Plugin Options

| Plugin | Package | Notes |
|---|---|---|
| **Official** | `@capacitor/push-notifications` | Returns native APNs token on iOS (not FCM token) — requires extra step for iOS FCM |
| **Community (Recommended)** | `@capacitor-firebase/messaging` | Unified FCM token on both platforms, handles swizzling, topic subscriptions, foreground notifications |
| **Community (Legacy)** | `capacitor-community/fcm` | Older; adds FCM on top of official plugin; less actively maintained |

**Recommendation for 2026:** Use `@capacitor-firebase/messaging` — it provides a unified cross-platform API and is the most actively maintained option.

### FCM Setup Steps
1. Create Firebase project; add Android app with your package name (e.g., `com.yourapp.couples`)
2. Download `google-services.json` → place at `android/app/google-services.json`
3. Add to `android/build.gradle`: `classpath 'com.google.gms:google-services:4.x.x'`
4. Add to `android/app/build.gradle`: `apply plugin: 'com.google.gms.google-services'`
5. Install plugin: `npm install @capacitor-firebase/messaging && npx cap sync`
6. Request notification permissions and register for push token in app startup

### Sideloaded APK Critical Constraint
- **FCM requires Google Play Services** on the device — this is a hard dependency
- Sideloading the APK itself is fine; the device just needs Google Play Services (GMS) installed
- Most Android phones outside China have GMS pre-installed — this is not an issue for standard Android devices
- **Exception:** Huawei HMS devices (post-2019 US sanctions), AOSP-only devices, or some Chinese market devices lack GMS → FCM will not work on these
- For GMS-less devices, you'd need HMS (Huawei Mobile Services) or a different push provider (e.g., Pushy.me, ntfy)

### Feasibility
- **Fully feasible** for sideloaded APKs on standard Android devices with GMS
- FCM delivery reliability on Android is high (~95%+) compared to iOS web push

## 4. vite-plugin-pwa (Vite 5+)

### Version Compatibility
- **From v0.17+**, `vite-plugin-pwa` requires **Vite 5**
- Current plugin versions (0.20.x+) are stable with Vite 5 and support the full PWA stack
- v0.20.2 introduced stricter behavior: throws error if `maximumFileSizeToCacheInBytes` Workbox warning is present

### Service Worker Strategies

#### `generateSW` (Default)
- Automatically generates a Workbox-powered service worker
- Pre-caches all static build assets
- **Limited push customization** — not recommended if you need custom push event handlers

#### `injectManifest` (Recommended for Web Push)
- You write your own service worker file
- Plugin injects the precache manifest at build time
- Allows full control over `push`, `notificationclick`, and `sync` event handlers
- Required for custom VAPID-based Web Push subscription logic

### Web App Manifest Generation
```js
// vite.config.js
VitePWA({
  registerType: 'autoUpdate',
  manifest: {
    name: 'Couples App',
    display: 'standalone',        // Required for iOS push!
    background_color: '#ffffff',
    icons: [{ src: '/icon-192.png', sizes: '192x192', type: 'image/png' }]
  }
})
```
The manifest is automatically generated and linked in `index.html`.

### Web Push Subscription Pattern (injectManifest)
```js
// Custom service worker (sw.js)
self.addEventListener('push', (event) => {
  const data = event.data?.json();
  event.waitUntil(
    self.registration.showNotification(data.title, {
      body: data.body,
      icon: '/icon-192.png',
    })
  );
});

self.addEventListener('notificationclick', (event) => {
  event.notification.close();
  event.waitUntil(clients.openWindow('/'));
});
```

### VAPID Key Handling
- Generate VAPID keys server-side (Django: use `pywebpush` or `py-vapid`)
- Client subscribes: `registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY) })`
- Send subscription object to your backend API for storage
- Backend sends push via Web Push Protocol to browser push service

### Service Worker Registration
```js
import { useRegisterSW } from 'virtual:pwa-register/vue' // or /react
// Automatically handles registration and update prompts
```

### Known Gotchas
- `display: 'standalone'` in manifest is **required for iOS push** — forgetting this silently breaks iOS push
- In dev mode, service workers are disabled by default; use `devOptions: { enabled: true }` for local testing (with caveats)
- Workbox `runtimeCaching` and push handlers are mutually exclusive in `generateSW` — use `injectManifest` for push

## Summary: Feasibility Assessment

| Area | Feasibility | Key Risk | Recommended Approach |
|---|---|---|---|
| iOS PWA Web Push | Moderate | Silent delivery failures after days | Use as best-effort; add WS/polling fallback |
| Django Channels JWT WS | High | Token exposure in query params | UUID ticket pattern via cache |
| Capacitor FCM Android (sideload) | High | Requires GMS on device | `@capacitor-firebase/messaging` plugin |
| vite-plugin-pwa Vite 5 | High | generateSW limits push customization | `injectManifest` strategy for Web Push |
