# Worker 1: Phase 2/5 Gap Analysis — Chat Push Notification Triggering

## Summary

Phase 5 specifies that "when a message is saved, send a push to group members who are NOT currently connected via WS" but provides no implementation. Two candidate integration points exist, each with distinct tradeoffs. Additionally, Phase 5 defines two helper functions (`remove_push_subscription`, `remove_push_subscription_by_token`) that are called in stubs but never defined.

---

## Integration Point Analysis

### Option A: Django Signal (`post_save` on `Message`)

```python
# chat/signals.py
from django.db.models.signals import post_save
from django.dispatch import receiver
from .models import Message

@receiver(post_save, sender=Message)
def trigger_chat_push(sender, instance, created, **kwargs):
    if created:
        from tasks.tasks import send_chat_push
        send_chat_push.delay(str(instance.id))
```

**Pros:**
- **Decoupling**: Notification logic is completely separated from WebSocket consumer. Any code path that creates a `Message` will automatically trigger pushes (REST API, admin, shell, tests).
- **Single source of truth**: The `Message.save()` path is the only trigger; no risk of consumer forgetting to call the push task.
- **Testability**: Signal can be tested in isolation with Django's `Signal.providing_args`.

**Cons:**
- **Hidden coupling**: Developers may not realize signals fire on every save. Can cause surprise "double push" if consumer also calls the task.
- **Tight coupling to model lifecycle**: If `Message` creation moves to a service layer or gets refactored, the signal must be kept in sync.
- **Global side effects**: Any future code that creates messages (e.g., message-importing scripts) will silently trigger pushes.

---

### Option B: Direct Call from `ChatConsumer.save_message()` After `group_send`

```python
# chat/consumers.py — inside receive(), after group_send:
async def receive(self, text_data):
    ...
    message = await self.save_message(user, self.group_id, content)
    await self.channel_layer.group_send(
        self.room_group_name,
        {'type': 'chat_message', ...}
    )
    # Trigger push for offline members
    from tasks.tasks import send_chat_push
    send_chat_push.delay(str(message.id))
```

**Pros:**
- **Explicit**: The push trigger is visible in the same function that creates the message. No hidden magic.
- **Lower latency**: Push scheduling happens immediately after the broadcast with no signal dispatch overhead.
- **Simpler to reason about for WS-specific behavior**: Chat is inherently tied to the WebSocket consumer flow in this architecture.

**Cons:**
- **REST API blind spot**: If a future REST endpoint creates messages (e.g., `POST /api/groups/<id>/messages/`), pushes won't fire unless explicitly added. This creates an inconsistency between WS and REST message creation.
- **Tighter coupling**: Consumer code is now responsible for async notification dispatch, mixing I/O responsibilities.
- **Duplication risk if extended**: If other code paths add message creation, someone must remember to also call `send_chat_push`.

---

### Recommendation: Signal-Based (Option A)

The **signal approach is cleaner** for this architecture because:

1. The `ChatConsumer` in Phase 3 uses `@database_sync_to_async` for `save_message()` — meaning message creation is already decoupled from the async consumer flow. Adding push logic directly in the consumer would mix notification concerns into the consumer.
2. Phase 3's `post_save` signal integration point already exists conceptually in the architecture (Phase 3 mentions signals and `auto_now_add` on `created_at`).
3. Phase 2's `send_task_reminder` already uses `ws_presence:{user.id}` Redis keys to skip users with open WS connections — the same mechanism applies here.
4. The risk of "double push" is mitigated by ensuring `send_chat_push` only sends to users **without** a `ws_presence:{user.id}` key (Phase 5 already specifies this check).

The consumer should **not** call `send_chat_push` directly; the signal handles it.

## Missing Helper Functions

Phase 5's `tasks/utils.py` stubs reference two helper functions that are never defined:

### `remove_push_subscription(endpoint)`

**Purpose**: Delete a `PushDevice` row when a Web Push subscription is expired (HTTP 404 or 410 from `webpush` call).

**Expected signature**:
```python
def remove_push_subscription(endpoint: str) -> None:
    """
    Remove a PushDevice row by its webpush subscription endpoint.

    Called when webpush() returns WebPushException with status 404 or 410,
    indicating the subscription has been invalidated server-side.

    Args:
        endpoint: The subscription endpoint URL (unique per PushDevice row).
    """
    PushDevice.objects.filter(subscription__endpoint=endpoint).delete()
```

**Notes**:
- Requires `subscription` field on `PushDevice` to store the full JSON subscription object (as JSONField or TextField with JSON serialization).
- Phase 2's subscribe endpoint stores `subscription` as a JSON field, so this query pattern is feasible.
- Should use `delete()` with no fallback — if subscription is gone, there's nothing to clean up.

---

### `remove_push_subscription_by_token(token)`

**Purpose**: Delete a `PushDevice` row when an FCM token is unregistered (`UnregisteredError` from Firebase SDK).

**Expected signature**:
```python
def remove_push_subscription_by_token(token: str) -> None:
    """
    Remove a PushDevice row by its FCM registration token.

    Called when FCM's messaging.send() raises UnregisteredError,
    indicating the token is no longer valid.

    Args:
        token: The FCM registration token (stored in PushDevice.fcm_token).
    """
    PushDevice.objects.filter(fcm_token=token).delete()
```

**Notes**:
- `fcm_token` field is already specified in Phase 2's `PushDevice` model.
- This is simpler than `remove_push_subscription` since tokens are direct string fields.
- Should also be safe to call even if the row doesn't exist (no-op).

## Celery Task `send_chat_push` Specification

For completeness, the task that should be wired from the signal:

```python
@shared_task
def send_chat_push(message_id: str):
    """
    Send push notifications to group members who are NOT currently connected via WebSocket.

    Reads ws_presence:{user.id} Redis keys (written by ChatConsumer on connect/disconnect)
    to determine which users should receive a push.

    Args:
        message_id: UUID of the newly created Message.
    """
    from chat.models import Message
    from users.models import GroupMembership, PushDevice
    from django.core.cache import cache

    try:
        message = Message.objects.select_related('group', 'sender').get(id=message_id)
    except Message.DoesNotExist:
        return

    # Get all group members
    memberships = GroupMembership.objects.filter(group=message.group).select_related('user')
    for membership in memberships:
        user = membership.user
        # Skip the sender — they sent the message and are already connected
        if user.id == message.sender_id:
            continue
        # Skip users with active WS connections
        if cache.get(f'ws_presence:{user.id}'):
            continue
        # Send push to all their devices
        devices = PushDevice.objects.filter(user=user)
        for device in devices:
            if device.type == 'webpush':
                send_webpush(device.subscription, ...)
            elif device.type == 'fcm':
                send_fcm(device.fcm_token, ...)
```

## Key Risk Mitigations

1. **Double-push prevention**: `send_chat_push` checks `ws_presence:{user.id}` before sending — users with open WS connections are skipped. This is consistent with Phase 2's `send_task_reminder` behavior.

2. **Sender skip**: The sender is explicitly excluded since they just sent the message.

3. **Error isolation**: If push sending fails for one user (non-expired subscription), it does not affect other users.
