# Worker 1: Redis Configuration Inconsistency Analysis

## Issue Summary

Phase 3 (Channel Layer) and Phase 6 (docker-compose) have a Redis configuration inconsistency that would cause the Django Channels WebSocket system to fail at runtime.

## Problem Details

### Phase 2 Redis Layout (plans/phase_2_plan.md, lines 136-138)
Phase 2 establishes the intended Redis database layout:
```
- REDIS_URL (default redis://127.0.0.1:6379/0): Celery broker and result backend
- REDIS_URL_CACHE (default redis://127.0.0.1:6379/1): Django cache and Channels channel layer
```

### Phase 3 Channel Layer Configuration (plans/phase_3_plan.md, line 30)
Phase 3's CHANNEL_LAYERS configuration uses:
```python
CHANNEL_LAYERS = {
    'default': {
        'BACKEND': 'channels_redis.core.RedisChannelLayer',
        'CONFIG': {
            'hosts': [env('REDIS_URL', default='redis://localhost:6379/1')],
        },
    },
}
```

**BUG**: Phase 3 references `REDIS_URL` instead of `REDIS_URL_CACHE`. This is inconsistent because:
1. Phase 2 intends for Channels to use `REDIS_URL_CACHE` (DB/1)
2. `REDIS_URL` defaults to DB/0, not DB/1
3. The default `redis://localhost:6379/1` in Phase 3 is misleading - if `REDIS_URL` env var is set, it would override this default entirely

### Phase 6 docker-compose (plans/phase_6_plan.md, lines 220-221)
```yaml
environment:
  REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
```

**MISSING**: `REDIS_URL_CACHE` is not defined in docker-compose. Only `REDIS_URL` is set to DB/0.

## Runtime Failure Scenario

1. In production (docker-compose), only `REDIS_URL=redis://:${REDIS_PASSWORD}@redis:6379/0` is set
2. Phase 3's `env('REDIS_URL', default='redis://localhost:6379/1')` resolves to `redis://:${REDIS_PASSWORD}@redis:6379/0` (DB/0)
3. But Celery also uses `REDIS_URL` for broker/backend (DB/0)
4. Result: Channels and Celery compete on the same Redis database, causing conflicts

## Resolution

### Phase 3 Fix (plans/phase_3_plan.md)
Change line 30 from:
```python
'hosts': [env('REDIS_URL', default='redis://localhost:6379/1')],
```
to:
```python
'hosts': [env('REDIS_URL_CACHE', default='redis://localhost:6379/1')],
```

### Phase 6 docker-compose Fix (plans/phase_6_plan.md)
Add `REDIS_URL_CACHE` to both `django` and `celery_worker` services:

```yaml
environment:
  DATABASE_URL: postgres://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}
  REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
  REDIS_URL_CACHE: redis://:${REDIS_PASSWORD}@redis:6379/1
```

### Phase 2 Integration Section Fix (plans/phase_2_plan.md, line 293)
The Phase 2 integration text says:
> "Django Channels Channel Layer (configured in Phase 3, also using Redis database `/1`)"

This is correct intent, but Phase 3's implementation doesn't match. After fixing Phase 3 to use `REDIS_URL_CACHE`, the integration text will accurately reflect reality.

## Correct Environment Variable Usage

| Component | Should Use | Database |
|-----------|-----------|----------|
| Celery Broker/Result | `REDIS_URL` | /0 |
| Django CACHES | `REDIS_URL_CACHE` | /1 |
| Channels Layer | `REDIS_URL_CACHE` | /1 |

## Verification

At runtime with the fixes applied:
- Django settings would resolve `REDIS_URL_CACHE` from environment
- If not set, falls back to `redis://localhost:6379/1` (local dev default)
- Channels connects to DB/1, separate from Celery's DB/0
- No Redis database contention between components
