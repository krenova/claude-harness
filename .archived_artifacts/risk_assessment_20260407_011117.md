### Open Questions Requiring Human Input

**Q-A: `IsGroupAdmin` `has_permission()` always returns `False` — intentional?**
Phase 2's permission class is designed to block all collection-level access and only allow object-level checks. The comment says this is intentional. Just confirming you are aware: this means a GROUP_ADMIN cannot be checked at the list-view level (only on detail views). If you ever need to filter a group list to "groups where I'm admin," you'd need a separate permission or filter.

**Q-B: Who can delete a task? — Spec/code mismatch**
The Phase 2 spec says "`DELETE ...` — Delete task (`Role.GROUP_ADMIN` or task creator)." The DRF implementation only checks GROUP_ADMIN. The "or task creator" part is not in the code. Please pick:
- **Option 1 (recommend):** GROUP_ADMIN only — update spec to match code
- **Option 2:** GROUP_ADMIN + task creator — add `IsTaskCreatorOrAdmin` permission to Phase 2

**Q-C: `GET /api/blog/posts/<slug>/` — Draft post via URL guessing?**
The endpoint filters `is_published=True` in the queryset, but DRF's `get_object()` pattern means a user who guesses a draft slug might still receive it. Please confirm:
- **Option 1 (recommend):** Use safe `get_object_or_404(Post.filter(is_published=True), slug=...)` pattern — return 404 for unpublished
- **Option 2:** Accept that slug-guessing protection is "good enough" for this app
